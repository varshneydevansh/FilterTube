How FilterTube Works

FilterTube uses a hybrid approach to hide content across YouTube:

Data interception (Zero-flash when possible)
When YouTube loads content via JSON, FilterTube removes blocked items before they render.
DOM fallback (Safety net)
For SPA navigation, hydration, and recycled DOM nodes, FilterTube scans the page and hides blocked elements.

Channel Identifiers

UC Channel ID (recommended)
A stable identifier like UCxjr9iAaC5QaIbp5hlERHDQ. This is the most reliable way to block a channel.
@handle (supports unicode)
Handles like @감동메모리 or @BTS. FilterTube normalizes percent-encoding and unicode so handles match everywhere.
Legacy channel URLs
Some channels appear as /c/Name or /user/Name. FilterTube stores them as normalized c/name or user/name and maps them to a UC ID when possible.

Import & Export backups

Export
  • Full vs Active-only: Master (Default) can export all profiles; non-master is forced to “Export active profile only”.
  • Formats: Plain JSON or Encrypted; encrypted exports require a PIN at import.
  • Destination: Downloads/FilterTube Export/ with versioned filename.
Import
  • Merges channels/keywords; matching entries update, new ones append. Profiles in the file are restored as-is.
  • Encrypted imports prompt for the PIN. Admin-only actions (full restore) require unlocking Master.
  • Auto-backup settings and modes from the backup are applied per profile.
Auto-backup
  • Per-profile toggle; mode (Latest/History) and format (Plain/Encrypted/Auto) follow the active profile.
  • Backups save to Downloads/FilterTube Backup/. Encrypted backups require a verified PIN to create.
  • On Firefox, encrypted downloads use a fallback save flow so backup export does not fail when the downloads API refuses the file.
  • Popup and New Tab both respect lock state: controls disable when locked.

Profiles & PIN behavior

Profile types
  • Default (Master): Admin for full exports, profile management, and account policy.
  • Account: Independent profile; can have child profiles.
  • Child: Follows parent; UI gating uses locks/PINs.
Locks & PINs
  • Locked profiles disable toggles and auto-backup controls until unlocked.
  • Encrypted exports/backups need the profile’s PIN; Master unlock is required for admin actions.
  • Unlock is per-session in both Popup and New Tab.
Auto-backup & actions
  • New profiles inherit the creator’s auto-backup mode/format/enable state.
  • Scheduling runs on profile creation and mode/format changes only if auto-backup is enabled for that profile.
  • Export scope toggle is disabled/forced to active-only for non-master; enabled for Master.
Where to change things
  • Popup: Quick toggle changes; respects locks/PINs; auto-backup controls visible when enabled.
  • New Tab: Full profile management (create/rename/delete, PIN set/remove), export/import, auto-backup mode/format.

Nanah Sync

Three simple ways to use Nanah
  • Send this profile once: Best default. Pair, verify, send, and let the other device review the update.
  • Parent controls child: Best for source → replica trust. Pick the remote child profile once, then save the managed link.
  • Move full account: Use only for migration, reinstall recovery, or the wider account tree.
  • The Advanced section is still there if you need to tune role, scope, or merge/replace manually.
How the pairing flow works
  • Pair: Create a 4-character code or QR on one device and join from the other.
  • Verify: Confirm the same safety phrase on both devices before sending anything.
  • Send: Share only the scope you want: active, main, kids, or full.
  • Simplest mental model: Pair, verify, send once, then save trust only if you want later sessions to reuse relationship and policy.
Quick start for parents
  • Choose Parent controls child on the sender.
  • Connect once and confirm the same safety phrase.
  • Pick the child profile in Remote target profile on the sender side.
  • On the first managed connection, the child device may still need one local parent PIN/unlock step to authorize that locked child profile.
  • After you choose Apply + Save Managed Link, later sessions can reuse that target and policy instead of asking the child to reconfigure everything again.
What is private and what is not
  • The relay only helps both devices find each other. Think of it as a meeting place, not a place where your settings live.
  • After the secure handshake, your settings are meant to move directly from one device to the other.
  • FilterTube does not need a central sync account to read or store your Nanah settings traffic.
  • There is no product goal here around reading your synced rules or chasing user-count analytics from your private settings.
  • You should still trust only your own devices and confirm the same safety phrase before applying anything.
Device roles and trust
  • Peer: Best for one-off syncing between your own equal devices.
  • Source / Parent: Can become the trusted sender for future replica updates.
  • Replica / Kids device: Receives proposals and can optionally auto-apply from trusted source devices.
  • Saved trust: A trusted link does not create always-on background sync. It remembers the relationship and policy for the next live session.
Managed links on the first connection
  • An untrusted Source → Replica session still asks the receiving device what to do with the first incoming update.
  • The replica can apply once without saving trust, or apply and save a managed link for future updates.
  • Saving a managed link is what turns future source-side policy into something precise and repeatable.
Allowed scopes inside a managed link
  • Each trusted managed link can allow one or more scopes: active, main, kids, or full.
  • The default scope and the merge/replace policy live on the trusted link itself, not only in the live session.
  • If an incoming managed update does not match the saved allowed scopes, FilterTube blocks it instead of applying the wrong scope.
  • Managed replica links can now choose a target mode on the receiving device: Current active profile or Always this local profile.
  • The fixed target mode is the safe way to make B -> B work across devices without guessing by profile name.
PINs and encrypted backups
  • PIN checks stay local. Nanah can require the sender or receiver to unlock locally, but the PIN itself is not sent.
  • Encrypted backup is still the archive path. Nanah is the live device-to-device sync path.
  • Encrypted full-account backup can also carry trusted-device recovery data for same-device reinstall recovery, but normal/plain export does not.
  • Restoring trusted-device recovery data is explicit during encrypted import. It is not silently restored from ordinary backup files.
  • Parents can still keep stricter control by using PIN-locked profiles plus trusted managed links for replica devices.
Refresh, reconnect, and trusted links
  • A live Nanah session is not meant to survive a page refresh, browser restart, or tab close. Refresh ends the live pairing session.
  • Trusted links are saved locally so you do not lose the remembered relationship, but today you still start a fresh live session when reconnecting.
  • The trusted-device card now has a Reconnect shortcut that starts the next fresh session with the saved role/policy defaults.
  • Saved trust means a faster next session, not automatic background syncing with no live connection.
  • If you try to reuse the same pairing code on the same active device session, FilterTube now blocks that locally instead of silently burning the code.
  • If no fixed target is saved on the managed replica link, non-full sync still writes into the receiver's current active profile at receive time.
  • If a managed replica link is saved with Always this local profile, future managed updates land in that pinned local profile even when some other profile is active on the receiving device.
What backups include today
  • Current Nanah trusted-link records are stored in local extension storage, not inside the normal FilterTube export/import payload.
  • Normal/plain backup restores your profiles and filters, but not the saved Nanah trust graph.
  • Encrypted full-account backup can include trusted-device recovery data, and import asks whether to restore it for same-device recovery.
  • That recovery path is intentionally explicit and should not be used to clone one device identity onto another device.
Master, account, and child behavior
  • Default / Master: Owns full import/export, master policy, and the broadest account controls.
  • Independent account: Owns its own settings and can manage its own child profiles, but does not gain authority over parallel accounts.
  • Child profile: Should be treated as a managed surface. In Nanah it is replica-only, so it should receive parent/source updates rather than act as its own source device.
  • Locked vs unlocked child: A locked child profile stays replica-only. If the parent unlocks that child profile locally, FilterTube can send that child profile’s own snapshot to another device without turning it into a full admin surface.
  • Targeting today: Child restrictions still depend on the profile currently active in the Nanah UI session, and child managed links should usually save a fixed local target so parent updates stay tied to that child profile.
  • Parent-safe child targeting: FilterTube now blocks “send anyway” from a child profile when the other device has some different profile active. The intended parent flow is to save one managed Source → Replica link on the receiving child profile and choose Always this local profile.
  • Strict child protection: Managed child links can now force approval-needed reconnect, no auto-apply, fixed local target, and local unlock required as one higher-safety preset.
  • Child profiles also stop exposing broad backup/import/export/account-admin actions in Settings, and the unlocked child surface no longer allows profile rename/delete/PIN mutation or trusted parent-link editing from Accounts & Sync.
Parent control without giving kids the PIN
  • Kids should not learn or type the parent PIN for normal managed updates.
  • The right model is local authorization on the child device: trust the parent/source link once, then allow only the saved managed scopes and policies.
  • FilterTube already keeps PIN values local. Trusted child links can now choose either Require local unlock or Allow trusted updates while locked, and that permission is saved only on the child device.
What the relay does
Nanah’s relay is only there to help both devices meet and exchange connection setup data. It is not meant to be a place where FilterTube reads your synced rules. After the secure handshake, your settings are meant to move directly device-to-device. The underlying project is open here: github.com/varshneydevansh/nanah.

Whitelist Mode

What is Whitelist Mode?
Whitelist Mode flips traditional filtering: instead of hiding blocked content, it hides everything except content from whitelisted channels and keywords. Perfect for creating an "allow-only" feed.
How to Build Your Whitelist

FilterTube reuses the same workflow as blocklist building:

  1. Switch to Blocklist Mode: Start in blocklist mode to browse freely
  2. Add Channels via 3-dot Menu: Use YouTube's "Block Channel" menu on videos you want to allow
  3. Switch to Whitelist Mode: Toggle the mode switch - your blocked channels become your allowed channels
  4. Enjoy Clean Feed: Only whitelisted channels and keywords will appear
Whitelist Mode Logic

When in whitelist mode, FilterTube:

  • Hides by default: All content is hidden unless it matches whitelist rules
  • Checks channels: Shows content from channels in whitelistChannels list
  • Checks keywords: Shows content with titles matching whitelistKeywords patterns
  • Preserves functionality: Watch pages and playlists work normally for allowed content
All subscriptions management page
/feed/channels stays visible even in whitelist mode so you can still see and manage all subscribed channels instead of locking yourself out of that management surface.
Creator and channel pages
Whitelist mode is stricter on creator/channel pages now. FilterTube avoids generic “fail open” behavior there and makes a safer decision when YouTube recycles or delays card identity data.
Mode Switching
Toggle between Blocklist and Whitelist modes using the top bar controls. FilterTube provides confirmation dialogs when switching to prevent accidental data loss, and automatically migrates your current blocklist to become your whitelist.

YouTube Kids Integration

Passive Listening on YouTube Kids
FilterTube provides complete Channel & Keyword based blocking on YouTube Kids by passively listening to the 3-dot menu UI in YT Kids content cards ("Block this video" and "Block this channel").
YouTube's Native Blocking vs FilterTube
When parents block channels using YouTube Kids' native 3-dot menu, those blocks are managed by YouTube's system. FilterTube maintains its own separate blocklist for Kids Mode, giving you additional control and backup options.
Understanding Channel Blocking Behavior

Parents should understand YouTube's complex channel blocking system:

  • Channels blocked via YouTube Kids' native UI are managed by YouTube's system
  • FilterTube's passive listener creates entries in its own Kids Mode blocklist
  • If a channel doesn't appear after removing from FilterTube's list, it may still be blocked by YouTube's native system
  • Use YouTube's official unblocking process for native blocks: Unblock channels from YouTube using your linked parent account
Parent Account Integration
YouTube Kids blocking requires a verified parent account. Parents can block content from their child's YouTube Kids profile and share content from YouTube to their child's profile. See YouTube Kids Help Center for complete details.

Troubleshooting

"Failed to fetch" while blocking
If FilterTube already knows the channel UC ID, it can still persist the block even if fetching the logo/name fails. Try reloading if the UI doesn't update immediately.
Blocked channel still appears
YouTube sometimes reuses DOM nodes during SPA navigation. FilterTube re-scans and clears stale cached metadata, but a hard refresh can help confirm persistence.
Subscribed-channel import
Imported subscriptions are now captured more defensively across browsers. If the first pass looks incomplete, reopen the subscribed channels surface and let FilterTube re-capture the page data before trying again.
